# Pokemon Champions 構築提案プロトコル (HARD-GATE)

> **目的**: 構築提案で数字根拠を欠いた誤りを防ぐ。
> 動画偏重の感覚提案ではなく、DB の採用率データを必須参照する手順。

---

## 全体フロー

```
[Step 1] 環境把握
  └─ champs.pokedb / pokechamdb / yakkun から TOP30 と全体動向取得

[Step 2] 個別ポケのデータ収集
  └─ 提案する 6 体それぞれについて:
      - champs.pokedb 個別ページ (持ち物/特性/性格/技/努力値の採用率)
      - pokechamdb クロスチェック
      - yakkun 育成論ページで補強

[Step 3] 実装確認
  └─ implementation_status.md で持ち物・技がナーフ/未実装じゃないか確認

[Step 4] 動画 + X で「なぜ」を補完
  └─ クロコ/今日ポケch./ポケソル/KYOUPOKE/note 記事 24-72h 以内の動向

[Step 5] 構築提案テンプレを埋める
  └─ data_extraction_guide.md の YAML テンプレで 6 体分

[Step 6] 私の選択 vs 採用率 のチェック
  └─ 各選択が採用率 30%+ なら主流、未満ならマイナー扱いを明示

[Step 7] ユーザーに提案 (鮮度・根拠ソース必須明記)
```

## チェックリスト (構築提案前に全部 ✓)

### データ系

```
[ ] champs.pokedb の個別ページを 6 体分 fetch 済み
[ ] pokechamdb クロスチェック済み
[ ] yakkun で実装/未実装確認済み
[ ] 各ポケの持ち物採用率トップ 3 把握済み
[ ] 各ポケの技採用率トップ 6 把握済み
[ ] 各ポケの性格採用率トップ 3 把握済み
[ ] 努力値分布の最頻パターン把握済み
```

### 鮮度系

```
[ ] DB データの last_updated が 7 日以内
[ ] クロコ/KYOUPOKE/ポケソルの直近 1 週間動画タイトル確認済み
[ ] 構築論に関連する動画があれば字幕取得済み
[ ] note 検索で 24-72h 以内の構築記事チェック済み
[ ] X (Nitter) で 24-72h パッチ言及チェック済み
```

### 整合性系

```
[ ] 提案する持ち物 6 個 = 全部実装確認済み
[ ] 提案する技 24 個 = 全部実装/ナーフ反映確認済み
[ ] メガストーン × 1-2 で実装確認済み
[ ] 採用率 30% 未満の選択は「マイナー型」と明示
[ ] テラス前提の対面想定をしていない
[ ] ステロは Champions 半減仕様 (4倍弱点25%) を考慮
```

### 出力系

```
[ ] T3 鮮度マーカー (鮮度: <Xh) を冒頭に書いた
[ ] 一次ソース (Tier S+) を最低 3 件引用
[ ] 動向ソース (Tier S) を最低 1 件引用
[ ] 各メンバーの「なぜこの型か」を採用率% で説明
[ ] 役割対象マトリクス (NxM) を表で図示
[ ] 選出パターンを 3-5 通り例示
[ ] ハイパー級ユーザー向けの易しさ確保 (専門用語は初出時注釈)
```

## 主流型 vs マイナー型の判断基準

| 採用率 | ラベル | 提案時の表現 |
|-------|--------|-------------|
| 50%+ | 圧倒的主流 | 「ほぼ全採用」と表現、特に理由なくこれを採用 |
| 30-50% | 主流 | 「主流の選択肢の一つ」 |
| 15-30% | 準主流 | 「選択肢として有力」 |
| 5-15% | マイナー | 「敢えてのマイナー選択、理由を明示」 |
| < 5% | レア | 「ロマン枠、勝率より楽しさ重視ならアリ」 |

ハイパー級ユーザーには **採用率 30%+ の選択を推奨**。
マイナー型は「なぜ敢えてそれか」の根拠なしには提案しない。

## 動画 vs DB の使い分け

| 情報種別 | 一次ソース (信頼度高) | 補完ソース |
|---------|----------------------|----------|
| 使用率 TOP | DB 系 | 動画ティアの裏取り |
| 持ち物採用率 | DB 系 | (動画は感覚値) |
| 技採用率 | DB 系 | 動画解説で「なぜ」 |
| 性格採用率 | DB 系 | (動画ほぼ言及なし) |
| 環境メタ感覚 | 動画 (クロコ/ポケソル) | DB は数字のみ |
| 新興構築論 | note記事/動画 | DB は反映遅れ |
| 個別ポケ評価 | 動画 tier発言 | DB は使用率のみ |
| 実装/未実装 | yakkun (公式) / アルテマ未実装一覧 | 動画はバラバラ |
| ナーフ/バフ | yakkun changes.htm | X @nomaneko_world / @poketettei |

**鉄則**: 数字 (%) は DB、評価は動画、両者整合性をチェック。

## アンチパターン (やってはいけない)

### ❌ 動画だけで構築組む
```
悪い例: 「クロコ氏が S 級と言ったから採用」
良い例: 「クロコ氏が S 級と評価 + DB で使用率 1 位 + 採用率 99% さめはだ」
```

### ❌ DB データの last_updated を確認しない
```
悪い例: 7 日前の DB データで「現環境は…」と語る
良い例: 「DB 4/28 23:44 更新、X 直近 24h で大変動なし → 信頼度高」
```

### ❌ 採用率の低い型を主流扱い
```
悪い例: 「メガカイリューは物理いじっぱりがテンプレ」
       (実際は ひかえめ 70.8% / いじっぱり 15.5%)
良い例: 「メガカイリュー主流は ひかえめ CSぶっぱ (70%)。
        いじっぱり物理は 15% のサブ型」
```

### ❌ 実装確認せず未実装アイテムを提案
```
悪い例: 「カバルドンに ゴツゴツメット を持たせる」
       (ゴツメ未実装)
良い例: 「カバルドンに オボンのみ (採用率 67%) を持たせる」
```

### ❌ ナーフ前の感覚で技を選ぶ
```
悪い例: 「アシレーヌのムンフォ起点で C 下げ狙う」
       (ムンフォの C 下げ確率 30%→10% にナーフ)
良い例: 「アシレ ムンフォの C 下げは 10% で偶発、
        ハイドロ主軸推奨」
```

## 実例: ガブリアスの正しい提案手順

```
1. データ取得
   - champs.pokedb /pokemon/show/0445-00?rule=0
   - pokechamdb /en/pokemon/garchomp
   → 取得結果:
     - 持ち物: タスキ40.5% / スカーフ30.6%
     - 性格: ようき69.9%
     - 技: じしん98% / げきりん61% / ステロ51.7% / 岩石封じ43%
     - EV: H4/A252/S252 = 55.3%

2. 実装確認 (yakkun)
   - タスキ ✓ / げきりん ✓ / ステロ ✓ / 岩石封じ ✓

3. 動画補強 (クロコ 4/26 + ポケソル 4/28)
   - クロコ: ガブは S 級唯一
   - ポケソル: お盆ガブが激増 (ステロ + 岩石封じ起点)

4. 私の構築 (剣舞型) vs データ
   - 剣舞 → 採用率 5% 未満 (技 TOP6 圏外)
   - 「マイナー型」確定

5. 修正提案
   - 主流型: タスキ × ようき AS × じしん/げきりん/ステロ/岩石封じ
   - 「ステロ起点作り」役 = ガブの環境的な立ち位置
   - 剣舞型は所持ポケに合わせる場合のサブ案として併記
```

## ハイパー級ユーザー向けの追加配慮

```
[ ] 専門用語が出たら初登場時に括弧書きで説明
   例: 起点作り(=積み技を使うための盤面準備)

[ ] 各ポケに「役割キーワード」を 2 つだけ付与
   例: ガブ = 「起点作り役」「対面壊し役」

[ ] 選出パターンを 3-5 通りに圧縮 (それ以上は記憶できない)

[ ] 「迷ったらこの 1 つ」フォールバック選出を提示

[ ] タイプ相性は「弱点 4 つ」だけ覚えれば OK の形に圧縮
```
