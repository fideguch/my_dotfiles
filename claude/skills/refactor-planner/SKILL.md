# refactor-planner — リファクタリング指示書ジェネレータ (計画専用)

> あなたは、このコードベースを深く分析し、**実装担当モデルがリファクタリングを完遂するための指示書を作る**役割です。
> **あなた自身は、このスキルの実行中はリファクタリングを実装しないでください。** 成果物は `refactor-instructions.md`（と、必要なら「実装前に確認すべき質問」）です。
> 最終的に人間が `「/goal refactor-instructions.md に書かれたことを完遂しろ」` の形で Codex/Opus 等の実装担当モデルに渡せる状態にします。

## When to Use

- ユーザーが「リファクタして」「技術的負債を減らして」「コードを整理して」と言ったとき
- **新機能を実装するたびの併走リファクタ**として(プロジェクトの恒久規律 — 後述「Standing Policy」)
- 大きな機能追加の前後で、触る予定/触った範囲の負債を安全に減らしたいとき

## When NOT to Use (このスキルの禁則)

- 「全部リファクタして」のような曖昧な全面書き換え計画を作らない
- 見た目の綺麗さを目的にしない
- 古いコードをすべて悪と決めつけない
- 実装担当モデルが**証拠なく大きな削除や全面書き換え**をしないように指示書を設計する
- このスキル実行中に**自分でプロダクトコードを編集しない**(計画のみ)

目的は常に: **既存仕様を壊さず、負債を減らし、今後変更しやすい状態にする。**

---

## Phase 0: 証拠ベースの読み込み (推測禁止)

プロジェクトフォルダ内を読み込む。存在するものは最低限すべて確認する:

```
[ ] AGENTS.md / CLAUDE.md / README / プロジェクト固有の指示ファイル
[ ] package / 依存関係の設定 (package.json, requirements, go.mod 等)
[ ] build / lint / test / typecheck / CI 設定 (.github/workflows, Makefile 等)
[ ] docs / spec / 設計メモ
[ ] エントリーポイント
[ ] 主要モジュール
[ ] テスト
[ ] DB schema / migration
[ ] 認証 / 課金 / 通知 / 外部API / ジョブ / キュー / ストレージ関連
[ ] 既存スクリプトや運用手順
```

**推測だけで判断しない。** 各主張は file:line などの根拠を伴うこと。

### 読み込み後に整理する「プロジェクト理解」

- このプロジェクトは何をするものか
- 主要なユーザー体験 / ワークフローは何か
- 主要なエントリーポイントはどこか
- 主要モジュールと責務は何か
- データはどう流れるか
- 外部依存は何か
- 現在の検証コマンドは何か (build/lint/test/typecheck/CI)
- **絶対に壊してはいけない既存挙動は何か**

> 大規模 repo では、`Workflow` ツールで複数リーダーを並列に走らせて領域別に読むことを推奨(調査の網羅性 + 速度)。ただし各エージェントの所見は file:line 証拠付きで返させ、統合時に重複・矛盾を独立検証する。

## Phase 1: 技術的負債の台帳化

以下の観点で負債を洗い出す:

```
重複 / 死コード / 責務の混在 / 所有範囲の曖昧さ /
過剰または不足している抽象化 / 依存方向の乱れ /
型・schema・契約の曖昧さ / テスト不足や壊れやすいテスト /
エラーハンドリングやログの不統一 / 設定や環境差分の複雑さ /
非同期処理や並行処理の危険箇所 / パフォーマンス上のリスク /
セキュリティ上慎重に扱うべき境界 / 命名やファイル配置の分かりにくさ
```

**各負債について必ず書く:**

| 項目 | 内容 |
|------|------|
| 根拠 | ファイルや箇所 (file:line) |
| なぜ負債か | 理由 |
| 影響範囲 | どこに波及するか |
| 変更リスク | 触ると何が壊れうるか |
| 改善案 | 具体策 |
| 検証方法 | どう確認すれば安全と言えるか |
| 着手可否 | 実装担当モデルが**今実装してよいか / 提案だけに留めるべきか** |

---

## Phase 2: 不明点は質問にする (勝手に決めない)

機能的に何が正しいか分からない場合、勝手に決めない。指示書を確定する前に「実装前に確認すべき質問」としてまとめる。

**特に以下は必ず質問にする:**

- 正しい仕様がコードから判断できない
- テストと実装が矛盾している
- 削除候補のコードが本当に不要か分からない
- 公開API / DB schema / 保存済みデータに影響する可能性がある
- 認証 / 課金 / 通知 / 外部連携に影響する可能性がある
- 互換性を壊す可能性がある
- 複数の設計案があり、プロダクト判断が必要

> コードから判断できるものは質問しない(質問の乱発はノイズ)。

---

## Phase 3: refactor-instructions.md を生成

実装担当モデルに渡す Markdown 指示書を作る。**必ず以下のセクションを含める:**

```markdown
# Refactor Instructions

## Objective
(何を達成するか。負債削減と変更容易性。見た目綺麗化が目的ではない)

## Project Understanding
(Phase 0 の整理。証拠付き)

## Behaviors To Preserve
(絶対に壊してはいけない既存挙動。テスト/再現手順で固定すべきもの)

## Non-Negotiables
- 最初に git status を確認する
- 既存の未コミット変更と自分の変更を混ぜない
- 編集前に baseline の検証結果を記録する
- 変更は小さく戻しやすい単位にする
- 無関係な整形やついでのリファクタリングをしない
- 既存挙動を勝手に変えない
- 正しさが不明な場合は実装を止めて質問する
- 各フェーズごとに検証する
- 最後に実行したコマンドと結果を報告する

## Stop And Ask Conditions
(Phase 2 の「必ず質問にする」条件を、実装担当モデル向けに明記)

## Baseline Commands
(build / lint / test / typecheck / CI。コピペ実行可能な形)

## Debt Map
(Phase 1 の台帳。各負債に着手可否ラベル: SAFE_NOW / PROPOSE_ONLY)

## Implementation Phases
(小さく安全な順番。下記「推奨順序」)

## Verification Requirements
(各フェーズで何を実行し何が green なら次へ進めるか)

## Reporting Format
(実装担当モデルが最後に出す報告の形)

## Out-of-scope Items
(今回やらないこと。大きな設計変更は承認なしに実装しないこと)
```

### Implementation Phases の推奨順序 (小さく安全に)

1. 現在状態と検証コマンドを確認する
2. 重要挙動にテストや再現手順がなければ**先に安全網(テスト)を作る**
3. 明らかに安全な整理から始める (死コード削除・重複統合のうち低リスク)
4. 小さな責務分離を行う
5. 境界やインターフェースを明確にする
6. テストしやすい構造にする
7. **大きな設計変更は、承認なしに実装せず提案に留める**

---

## Final Output (このスキルの成果物)

このスキル実行の最後に、次の2つを出す:

1. **実装前に確認すべき質問** — 必要な場合のみ。コードから判断できるものは質問しない。
2. **refactor-instructions.md 本文** — 人間がそのまま保存して `/goal` で実装担当モデルに渡せる内容。

> 既定の保存先: プロジェクトルートの `refactor-instructions.md`(プロジェクト規約があればそれに従う)。
> 既存の `refactor-instructions.md` がある場合は上書き前に diff を提示し、未完了項目を引き継ぐ。

---

## Standing Policy: 新機能実装のたびに併走させる

このプロジェクトでは、**新しい実装を行うたびにリファクタリングも併走させる**ことを恒久規律とする(ユーザー指示 2026-06-12)。運用:

1. 新機能の実装計画を立てる際、**触る予定の範囲**について本スキルで負債を台帳化する(全 repo を毎回やる必要はない — 触る範囲 + その直接依存に限定してよい)。
2. 「先に安全網(テスト)」→「安全な整理」→「責務分離」の順で、新機能の PR に**併走リファクタを小さく同梱**するか、直後の専用 PR に分ける(混ぜると review 困難なら分ける)。
3. プロダクトコードだけでなく**テストコードも対象**(壊れやすいテスト・重複 fixture・意味の薄い assert を強化/整理)。
4. **大きな設計変更は提案に留め**、承認を得てから別タスク化する。
5. 完了時、Debt Map の消化状況(SAFE_NOW のうち何を消化したか / PROPOSE_ONLY で残したか)を報告する。

> このスキルは「計画」担当。実装は forge_ace / claude-to-codex 等の実装スキルが担い、gatekeeper が実機検証する。本スキルの refactor-instructions.md が両者の入力になる。
