# 配布資料 デザイン規約 v2（A4縦・モノクロ・ドキュメント型）

> これはテンプレートです。プロジェクト導入時に `slides/DESIGN.md` としてコピーし、案件固有欄を埋めて確定させてください。

本プロジェクトの配布資料はすべて本規約に従う。根拠は組版・文書設計のリサーチ（W3C日本語組版処理の要件、モノクロ印刷の濃度規範、Amazonナラティブ文書、CSS Paged Media定石）に基づく。

## 0. これは何のための文書か（最重要・全設計の前提）

- **読者は〔クライアントの読者像〕（クライアント・〔読者の属性〕）**。用途は2つ：①会議（〔会議形式〕）で出口を順を追って説明する台本になる ②読者が後日ダウンロードして**一人で読み返したときに、話者なしで理解でき、読む価値がある**。
- したがってこれは**プレゼンスライドではなく「読む文書」**（Amazonのナラティブメモの発想）。スライドと文書は要件が違い、兼用は両方失敗する（slideument批判）——本資料は最初から文書側に振り切る。
- 旧規約の「箇条書き5点まで」「1枚1メッセージ」は**廃止**。各ページに文脈のあるちゃんとした文章を書く。箇条書きは「列挙が本質」のときだけ使い、説明は散文で書く。

## 1. 制作工程（厳守・順序を飛ばさない）

1. **構成MD**を書く：ページ一覧→各ページの**ワイヤーフレーム**（配置の設計）→内容の当て込み。`slides/NN_名前.md`
2. **HTML**に変換：`slides/NN_名前.html`（自己完結・単一ファイル）
3. **PDF（またはPPTX）**に出力（指示があったときのみ）
※いきなりHTMLやスライド本体を作ることを禁止する。

## 2. 判型と版面（A4縦）

| 項目 | 規定 | 根拠 |
|---|---|---|
| 判型 | **A4縦** 210×297mm | 情報量を詰められ、印刷・PDF閲覧の標準 |
| 余白 | 上25mm・左右20mm・下20mm（`@page { size: A4 portrait; margin: 25mm 20mm 20mm; }`） | 端2cm確保の定石＋天にヘッダー余地 |
| 本文幅 | 170mm・**1カラムのみ**（2カラム禁止） | 日本語約42字/行＝可読上限内。会議での指差し説明を優先 |
| 画面表示 | グレー地（#f0f0f0）に白い210mmページを影付きで縦に並べる。印刷時（@media print）は影・地色を除去 | CSS Paged Media定石 |
| ページ番号 | 各ページ下部中央「n / 総数」＋フッター左に資料名（回） | 文書として参照可能に |

## 3. 組版（文字）

| 要素 | 規定 |
|---|---|
| フォント | "Noto Sans JP"（Google Fonts）→ 予備 "Hiragino Kaku Gothic ProN","Yu Gothic",sans-serif。等幅は "Noto Sans Mono",monospace |
| 本文 | **15px（印刷11pt相当）／line-height 1.75** |
| 見出し | H1（ページ題）24px・H2 19px・H3 16px（ジャンプ率1.3〜1.6）。見出しの直後で改ページしない（break-after: avoid） |
| 小さい文字 | 最小12px（キャプション・フッターのみ）。line-height 1.5 |
| 太字 | **見出しと、そのページの結論1〜2箇所のみ**。乱用禁止（全部太字＝強弱ゼロ） |

## 4. モノクロ規則（色は使わない）

- 使ってよいのは**黒とグレーの階調のみ**：本文 #1a1a1a／補足 #555／罫線 #999（1px実線）／背景グレー #e9e9e9（約12〜15%網。**10%未満は印刷で消えるため禁止**）。
- 小さい文字・細い線は純黒で（薄グレーの小要素はかすれる）。
- **強調の序列**（この順で使い、これ以外の装飾をしない）：
  1. 見出し（サイズ＋太字）
  2. ページ冒頭の**要旨**（後述）
  3. 結論の太字（1〜2箇所）
  4. 枠囲み（背景 #e9e9e9＋左罫線3px黒）＝そのページで持ち帰るべき要点・注意
  5. 罫線＝区切り・表
  6. 余白＝それ以外の階層はすべて余白と行間で作る
- 完成後、そのままグレースケールで階層が成立していることを確認する（元からモノクロなので、確認は「強調過多になっていないか」）。

## 5. ページの型（各ページ共通の骨格）

```
┌──────────────────────────┐
│ 章ラベル（小・グレー）                    │
│ ページ見出し（H1）                       │
│ ■要旨: このページの結論を1〜2文（太字）    │ ← 必ず置く。黙読でもここだけで流れが追える
│ ──────────────────────── │
│ 本文（散文中心。図・表・枠を必要に応じて）   │
│                                        │
│ （下部が余っても埋めない）                 │
│ フッター: 資料名・回        n / N        │
└──────────────────────────┘
```

- **縦方向は上から自然に流す**。1ページ内の均等配置はしない。ページ下部の余白は正常（埋めるための引き伸ばし禁止）。
- 縦リズム：余白・間隔は8pxの整数倍で統一。見出しの前余白：後余白＝2：1。
- 表・図・枠・プロンプトブロックは `break-inside: avoid`（分断禁止。1ページに収まらない長さの場合のみ、切りの良い位置で明示的に分割）。

**記入欄の禁止と【決定事項】マーカー**
- 資料に手書き・入力する運用はしない。**実施日欄・記入枠・下線の空欄・チェック記入欄（合意用）は作らない**。
- 会議で合意が必要な箇所は、枠囲み（§4の序列4）に **【決定事項】** ラベルを付けて「ここで決める」ことを明示するだけにする。
- 決まった内容は、会議後に発注者がClaudeへ伝える → 構成MDへ反映 → HTML再生成、という流れで**資料そのものを最新の合意内容に更新**する（配布資料が常に唯一の正）。
- 担当者名・個人名は資料に書かない。

**チェックリスト・確認手順の書き方（技術知識ゼロ基準）**
- チェックリストは飛ばし読みされる前提で書く。**初出定義に頼らず、各項目の中で完結**させる。
- 各項目は「**どこを見て → 何が確認できたらOKか → なぜ確認するのか**」の3要素が1項目内で伝わる書き方にする（なぜ、は——や（）で短く添える）。
- 技術用語（設定ファイル名・専門ツール名等）を項目内で使う場合は、その場で（＝一言説明）を必ず添える。ファイル名だけを裸で書かない。
- 判定基準: **技術知識が全くない読者が、この1項目だけを読んで1人で実施できるか**。できない項目は書き直す。
- 確認に失敗した場合の逃げ道（例:「入ってしまっていたらClaudeに削除の手順を聞く」）を必要に応じて添える。

## 6. 図・表

- 図はモノクロの線画（線 #1a1a1a、塗りはグレー階調のみ）。HTML/SVGで描く。矢印・箱・ラベルで構成し、装飾しない。
- 表は罫線1px・ヘッダー行は背景 #e9e9e9＋太字。列数は4以下、それを超えるなら分割。
- 図表には番号とキャプション（12px・グレー）を付ける。

## 7. プロンプトブロック（本資料の最重要部品）

「Claudeに貼れば実装できるプロンプト」を掲載する部品。**レイアウトの美しさより、全文が欠けず表示されること・コピーしやすいことを最優先**する。

| 項目 | 規定 |
|---|---|
| 表示 | `white-space: pre-wrap; overflow-wrap: break-word;` で**全文折返し表示**。横スクロール・切り捨て・省略は禁止 |
| 書体 | 等幅13px／line-height 1.6。背景 #f5f5f5・枠線1px #999 |
| コピー | ブロック右上に「コピー」ボタン（Clipboard API。押すと「コピーしました」に2秒変化）。**印刷時は非表示**（@media print） |
| 見出し | ブロック直前に「プロンプト◯：（目的）」と、使い方1〜3行（どこに貼るか・貼る前に何を用意するか） |
| 改ページ | 原則 break-inside: avoid。1ページ超の長さは「（続き）」見出し付きで明示分割 |
| 内容 | 構成MDの本文から**一字一句変えずに**転記する（HTML化での要約・整形禁止） |

## 8. ツールURLの規定（プロジェクト全体で厳守）

- 導入・操作するツール（GitHub、Supabase、Claude等）は、**目的画面への直URLを本文の初出箇所に明記**する（例: 新しいリポジトリの作成 → https://github.com/new ）。「こちら」等のリンクテキスト置換・短縮URL・省略は禁止。URLは等幅書体で全文表示する。
- **資料の最終ページに「本資料のURL一覧」を必ず置く**。読者が資料に戻れば、URLをコピーするだけで必ず目的の画面にたどり着けることを保証する（検索して迷子にさせない）。
- PDFでもコピーできるよう、URLをボタンや画像にしない。

## 9. 機密・共通ルール

- クライアントの顧客の実名・実データ・APIキー・個人名を含むパスは一切載せない（例示は「顧客A」「練習用」等の一般名）。
- 担当者名は書かない。口頭対応と決めた事項は資料に書かない。
- 業務用語はクライアントのアプリ・現場の画面表記と統一する。専門用語は初出で一言定義。
- ファイル名：`slides/NN_名前.md` → `slides/NN_名前.html`（→ 指示時のみ `.pdf`）。
- （案件固有）用語統一: 〔案件固有の用語A〕／〔案件固有の用語B〕／〔案件固有の用語C〕。「〔用語〕」＝〔定義〕。〔資料に掲載しない固有事項〕は資料に書かない。〔転用時: クライアントの現場用語・固有ルールに置き換える〕

## 10. 納品前チェックリスト

- [ ] 全ページに「要旨」があり、要旨だけ拾い読みして流れが通る
- [ ] 話者なしで読んで理解できる（文脈の欠けた断片がない）
- [ ] モノクロのみ・強調の序列（§4）を守っている／太字の乱用がない
- [ ] **記入欄・空欄・担当者名がない**。合意箇所は【決定事項】マーカーで明示されている
- [ ] チェックリストの全項目が「どこを見て・何が確認できたらOKか・なぜ」を含み、技術知識ゼロの読者が1人で実施できる（用語のその場補足あり）
- [ ] 登場するツールの直URLが初出箇所にあり、**最終ページにURL一覧がある**
- [ ] プロンプトブロックが全文表示・コピー可能・MDと一字一句一致
- [ ] A4印刷プレビューで表・図・枠・プロンプトの分断がない
- [ ] 機密（実名・実データ・キー・口頭対応事項）が載っていない
- [ ] ページ番号・フッターがある

## 11. 標準構成テンプレート（転用ガイド）

本規約は案件非依存。転用時は §9 の「案件固有」行だけ差し替えれば、他案件でも同じ品質の資料が作れる。

**キックオフ資料の標準構成（6〜7ページ）**
①表紙（タイトル＋本日の流れ）→ ②現状の確認（クライアントの実務で対象がどう機能しているか）→ ③仕組みの全体図（1枚図＋読み解き文）→ ④所感（できていることの明文化＋総評）→ ⑤目標と達成の定義（【決定事項】）→ ⑥ロードマップ（【決定事項】）→ ⑦決めごと・次回までのお願い

**レクチャー回資料の標準構成（7〜10ページ）**
①表紙（タイトル＋本日の流れ）→ ②前回の振り返り（**確定済みの決定事項を反映した状態で配布**）→ ③今回やること（2件まで・各1段落）→ ④テーマごとに〔説明ページ → 実行ページ（プロンプト／手順＋完了確認チェック）〕→ ⑤宿題（基本形は「資料だけを見て自分の環境でもう一度再現する」）＋次回予告 → 最終ページに「本資料のURL一覧」

**制作・更新サイクル**
構成MD作成（ワイヤーフレーム→内容）→ HTML → （指示時のみ）PDF。会議で決まった内容は構成MDに反映して再生成し、常に最新の合意を反映した資料を唯一の正とする。

〔転用時: クライアントの現場用語・固有ルールに置き換える〕
